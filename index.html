Module 00 · Start here

Two ways to talk to a computer

You already know one way: you point and click. This course teaches the other way — typing commands — by walking through the real setup of a small app, mistakes and all.

By the end of this module you'll know

  • The difference between clicking and typing commands
  • Why typed commands exist when buttons feel easier
  • What project we'll set up, and how the course is organized

Most of the time you tell a computer what to do by pointing at pictures: double-click an icon, drag a file, tap a button. That's a graphical interface, and it's friendly — but you can only do the things someone drew a button for.

The other way is to type words. You type a command, press Enter, and the computer does it and types back. It's a text conversation instead of a point-and-click one. That's what this whole course is about, because it's how nearly every developer tool expects to be used.

🗣️
Think of clicking as ordering from a picture menu — easy, but limited to the photos on the wall. Typing commands is speaking the language — once you know a few words, you can ask for anything.

Why bother, if buttons are easier?

Three reasons worth keeping in mind as you go:

  • Precision. A command says exactly what you want, with no ambiguity.
  • Automation. Typed commands can be saved, repeated, and chained. You can't easily record "click here, then here" — but you can save a list of commands and re-run it forever.
  • It's where the tools live. Python, pip, and the kind of app we'll install were built to be driven by typed commands. Often there is no button — the command line is the front door.
Our running project

Throughout the course we'll set up "Danvion", a small dashboard app that tracks how long you spend in each program on your computer. You don't need to understand its code — it's just a realistic excuse to learn the setup skills every project shares: navigate to a folder, install some packages, and run it.

How this manual works

Seven short modules, in order. Each one has a clear goal, plain-English explanations with analogies, real commands, and a quick knowledge check. There's a hands-on practice terminal in Module 02 you can type into safely, plus a glossary and a printable cheat sheet at the end. Mark each module complete to fill your progress bar.

Module 01 · The tools

The terminal and the shell

The "black window where you type" has two parts with two names. Getting them straight now will make every later step easier to follow.

By the end of this module you'll know

  • What a terminal is vs. what a shell is
  • The common shells: bash, zsh, PowerShell, Command Prompt
  • How to open a terminal on Windows, macOS, and Linux

Terminal = the window. Shell = the worker inside it.

It helps to split the thing in two:

  • The terminal is the window — the app that gives you a place to type. On macOS it's literally called "Terminal"; on Windows it's "PowerShell," "Command Prompt," or "Windows Terminal"; on Linux it's often "GNOME Terminal" or "Konsole."
  • The shell is the program running inside that window that actually reads your commands and carries them out. Common shells are bash, zsh (macOS's default), PowerShell, and cmd (Command Prompt).
☎️
The terminal is the telephone handset; the shell is the person on the other end who understands what you're asking for. You speak into the handset, but it's the person who actually does things.

For most of this course you don't need to keep them perfectly separate — "open a terminal" just means "open that window where I can type." But the distinction matters in Module 03, where the specific shell you're using (PowerShell vs. Command Prompt) changes what works.

Deep dive: why does this even exist?

The terminal is older than windows-and-icons. In the 1970s, before graphical screens existed, people talked to computers entirely through text on a printer-like device called a teletype — that's where the abbreviation "tty" you sometimes see comes from. The graphical desktop was layered on top later. The command line stuck around because it's so precise and automatable, not because anyone forgot to replace it.

Knowledge check

You open "PowerShell" and type a command. Which part is the shell?

The shell is the worker that interprets commands (here, PowerShell-the-program). The window it appears in is the terminal. They share a name in this case, which is exactly why the two ideas are worth separating.

Module 02 · The core skill

Where am I? Folders, paths & cd

This is the single most important idea in the whole course. Once "which folder am I standing in?" clicks, half of all beginner errors explain themselves.

By the end of this module you'll know

  • How files are organized as a tree, and what the "current folder" means
  • pwd, ls/dir, and cd — your three core commands
  • Relative vs. absolute paths, and how to read two famous error situations

Your files are a tree, and you're standing in one branch

Your files aren't in one big pile. They live in a tree of folders inside folders. ("Folder" and "directory" mean the same thing — "directory" is just the older, more technical word.) Your home folder holds Downloads, which might hold a danvion folder, which holds app.py and a templates folder, and so on.

Here's the key part: in a terminal, you are always "standing" somewhere in that tree. The shell tracks one folder you're currently in, called the current working directory. Everything you type is interpreted relative to that folder unless you say otherwise.

🏠
The current working directory is the room you're standing in inside a big building. Say "grab the spoon" in the kitchen and you mean the kitchen's spoon — you don't recite the house's full address every time. The shell works the same way: commands act on the room you're in.

Three commands to look around and move

pwd means "print working directory" — it answers where am I?. dir (Windows) and ls (Mac/Linux) list what's in your current room — think of them as your flashlight. And cd means change directory: it's the command for walking from one room to another.

cd, and the two kinds of address

cd doesn't open, move, or copy anything — it just changes where you're standing, which changes what your other commands act on. Two ways to tell it where to go:

  • Relative path — read from the room you're in. cd danvion means "from here, step into danvion." cd .. means "go up one level." (One dot, ., means "right here.")
  • Absolute path — the full address from the very top of the tree, so it works no matter where you are now: cd C:\Users\You\Downloads\danvion on Windows, or cd /Users/you/Downloads/danvion on macOS.
🧭
A relative path is "two doors down from where I'm standing." An absolute path is the full street address. cd happily accepts either.

Your turn: drive a real terminal

This is a safe, simulated Windows shell — it can't touch any real files. You start at the very top of the C: drive. Your mission: get into the danvion folder and run dir until you can see app.py. Try dir, cd, cd .., and pwd. Type help if you're stuck.

Two famous situations, now decoded

1. "dir listed my whole C: drive!"

That's dir working perfectly — it lists whatever room you're standing in. If it showed the whole drive (folders like Program Files, Users, Windows), it's telling you exactly where you are: the root of C:, the very top — the building's lobby. You simply haven't walked into your project folder yet. cd into it and run dir again; the listing shrinks to your folder's contents.

2. "The system cannot find the path specified."

This is the computer saying: "You named a path, but there's nothing there." Because paths are read relative to the room you're in, it almost always means one of two things: the folder doesn't exist from where you currently are, or you're standing somewhere unexpected. The fix is always the same flashlight: run dir (or ls) to see what's actually around you, then adjust.

Two time-saving tricks (Windows)

Skip typing long paths: type cd and a space, then drag the folder from File Explorer onto the terminal window — it pastes the full path for you.

Open a terminal already in the right room: open the folder in File Explorer, click the address bar, type powershell, press Enter. The terminal opens already standing inside that folder — no cd needed.

Knowledge check

You run dir and see the entire C: drive. What does that tell you?

dir always lists your current folder. Seeing the whole drive means your current folder is the top of the drive. Walk into your project with cd, then dir again.

Module 03 · A subtle trap

Running a script vs. opening it

A surprising number of "it asked me which app to use and Notepad popped open" moments come down to one distinction: running a file versus opening it as a document.

By the end of this module you'll know

  • Why a script can open in Notepad instead of doing anything
  • How to tell Command Prompt and PowerShell apart at a glance
  • What that scary block of gibberish at the bottom of a script is

Cooking the recipe vs. reading the recipe

A script file — like a .ps1 (PowerShell) or .bat (Command Prompt) file — is a list of instructions meant to be run by a shell.

📋
Running a script is cooking the recipe. Opening it (e.g. double-clicking, or letting Windows "open with…") just shows you the recipe written on paper — nothing gets cooked.

So when you double-click a .ps1 and Windows asks "which app do you want to use?", then opens it in Notepad — nothing went wrong and nothing bad happened. Windows just didn't know to run it, so it fell back to opening it as text. You saw the recipe, not the meal.

Why did that happen?

Usually because you were in Command Prompt, which doesn't speak PowerShell. Hand cmd a .ps1 and it shrugs and tries the "open file" route — triggering that app-picker dialog. The lesson: scripts get run by a terminal; double-clicking just shows you the text.

Which shell am I in? Read the prompt.

Command Prompt and PowerShell look similar but behave differently. Tell them apart by the very start of the prompt line:

prompts compared
:: Command Prompt — no prefix
C:\Users\You\danvion>

:: PowerShell — starts with PS
PS C:\Users\You\danvion>

That little PS is the tell. To run a .ps1 activation script you want PowerShell; the surest way to get it is the File-Explorer-address-bar powershell trick from Module 02. In Command Prompt, the equivalent script would end in .bat instead.

What was that wall of gibberish at the bottom of the script?

If you opened a real .ps1 and saw hundreds of lines of random characters under a line reading # SIG #, that's a digital signature — a tamper-proof stamp proving the file is the genuine, unmodified official version. It's not code you run, not malware, and nothing you ever need to read or edit. Safe to ignore completely.

Knowledge check

You double-click Activate.ps1 and it opens in Notepad. What happened?

Opening ≠ running. Notepad showed you the script's text. To actually run it, type it inside PowerShell (look for the PS prompt), rather than double-clicking.

Module 04 · The toolkit

Python, packages & pip

Our project is written in Python and relies on code other people wrote. Here's what that means, and the one command that fetches it all.

By the end of this module you'll know

  • What Python is, and python vs py on Windows
  • What a "package" is and why we don't reinvent the wheel
  • pip and requirements.txt: installing a project's dependencies

Python, and a Windows wrinkle

Python is a programming language and the program that runs Python code. You invoke it from the terminal by name. On macOS and Linux that's usually python3. On Windows it's python — except Windows has a common quirk:

Windows gotcha

Sometimes typing python on Windows pops open the Microsoft Store or says it isn't recognized. The more reliable command on Windows is py — the "Python launcher" that ships with the official installer. If python misbehaves, try py instead (e.g. py app.py).

Packages: standing on other people's shoulders

A package (or "library") is a bundle of pre-written code that solves a common problem, published so anyone can reuse it. Our app uses packages like flask (to run a small web server) and psutil (to read system info). Nobody re-writes those from scratch — you install them.

🧱
Packages are LEGO bricks other people molded. Instead of inventing a wheel-shaped brick yourself, you grab the one that exists and click it onto your build.

pip and the shopping list

pip is Python's package installer — the tool that downloads packages and sets them up. You can install one by name (pip install flask), but projects usually ship a requirements.txt file: a plain-text shopping list of every package the project needs.

install dependencies
PS C:\Users\You\danvion> pip install -r requirements.txt
Collecting flask...
Collecting flask-socketio...
Collecting psutil...
Successfully installed flask-3.x flask-socketio-5.x psutil-7.x

Read pip install -r requirements.txt as: "pip, install everything listed in this requirements file." The -r means "read the list from this file." One line installs the whole set.

Note

If pip isn't recognized, try pip3, or py -m pip install -r requirements.txt (which means "Python, run your pip tool"). This only works once you're standing in the folder that contains requirements.txt — back to Module 02's golden rule.

Then: run the app

With packages installed, you start the program with python app.py (or py app.py). For our project, that launches a small web server on your own machine and prints something like Running on http://127.0.0.1:5000. You then open http://localhost:5000 in a browser to see it.

What do "localhost" and "port 5000" mean?

localhost means "this very computer" — the server is talking only to you, not the internet. A port (here 5000) is like an apartment number at that address, so multiple programs can share one machine without mixing up their mail. Heads-up: on macOS, port 5000 is sometimes taken by AirPlay; if so, the app can be told to use 5001 and you'd open localhost:5001 instead.

Knowledge check

What does pip install -r requirements.txt do?

pip is the installer; -r requirements.txt points it at the project's shopping list of packages. It must be run from the folder that contains that file.

Module 05 · Good hygiene (optional)

Virtual environments

A "venv" gives one project its own private toolbox of packages. It's optional — but understanding it explains a confusing command and a common Windows error.

By the end of this module you'll know

  • The problem venvs solve, and why they're optional
  • python -m venv venv decoded (why "venv" appears twice)
  • Activation, the (venv) prompt, and the PowerShell execution-policy fix

The problem: clashing versions

By default, pip installs packages into one shared system-wide Python. That's fine until you have a second project. Imagine Project A needs version 1 of a library and Project B needs version 2 — they can't both live in the same shared Python without colliding.

🧰
A virtual environment is a dedicated toolbox for one project. Whatever you install goes in that box, can't break another project's tools, and if it ever gets messed up you just throw the box away and make a new one.

Creating one: why "venv" twice?

create the toolbox
PS C:\Users\You\danvion> python -m venv venv

The repetition trips everyone up. Here's each piece:

  • python — run Python.
  • -m — "run a built-in module by name," not a file of mine.
  • the first venv — the name of that built-in tool (Python's environment-maker).
  • the second venv — an argument: the folder name to create. You chose this; by convention it's venv, but python -m venv toolbox would make a folder called toolbox.

So: "Python, run your venv tool, and build a new environment in a folder called venv." Two different things that happen to share a name — one is the tool, one is the folder. You run this once, ever.

Activating it

Creating the toolbox doesn't make your terminal use it. Activating flips your current terminal over to the project's private Python.

SystemActivate command
Windows (PowerShell)venv\Scripts\Activate.ps1
Windows (Command Prompt)venv\Scripts\activate.bat
macOS / Linuxsource venv/bin/activate

Activation doesn't move any files — it just adjusts your current terminal so that python and pip point at the venv's copies. (Under the hood it puts the venv at the front of your PATH, the list of places the shell looks for programs.) You'll know it worked because your prompt gains a (venv) tag:

activated
(venv) PS C:\Users\You\danvion>

Type deactivate to step back out. Activation only lasts for that one terminal window — open a new one and you'd activate again (but you never re-create the venv).

The #1 Windows snag

PowerShell often refuses to run scripts by default — you'll see a red "running scripts is disabled on this system." That's its execution policy protecting you. To allow your own local scripts, run this once, then re-try activation:

Set-ExecutionPolicy -Scope CurrentUser RemoteSigned

In plain terms: "for my account only, allow scripts that are local or properly signed." It's a standard, reasonable setting.

Honest advice

The virtual environment is optional. If it's fighting you, skip it entirely — run pip install -r requirements.txt and python app.py directly, and the app works exactly the same. For a single small project, skipping it is a perfectly reasonable choice.

Knowledge check

In python -m venv venv, what is the second "venv"?

First venv = the built-in tool (run via -m). Second venv = the folder name you're asking it to create. You could have written any name there.

Module 06 · The whole workflow

Putting it all together

Every skill from the course, assembled into the real start-to-finish setup — plus a troubleshooting playbook for when something goes sideways.

By the end of this module you'll know

  • The complete setup sequence, in order
  • Why folder layout (the templates rule) matters
  • A fast lookup table for the common errors

The full sequence

  1. Arrange the files. Put app.py and requirements.txt in a folder, and place index.html inside a sub-folder named exactly templates. The app looks for that folder by name — get it wrong and the page won't load.
  2. Open a terminal in that folder (File Explorer → address bar → powershell). Confirm with dir that you see app.py.
  3. (Optional) Create & activate a venv: python -m venv venv then venv\Scripts\Activate.ps1.
  4. Install packages: pip install -r requirements.txt.
  5. Run it: python app.py (or py app.py). Leave the terminal open.
  6. Open http://localhost:5000 in your browser. Press Ctrl + C in the terminal to stop.

Appendix: platform notes for this specific app

Windows works out of the box. macOS reads the active app via the system, and the first run may prompt your terminal for Accessibility permission (System Settings → Privacy & Security → Accessibility) — grant it or app names come back blank. Linux (X11) needs sudo apt install xdotool xprintidle; on a Wayland session no app can read the active window, so values stay "Unknown." All usage data lives in a local usage_data.json file next to app.py — delete it to wipe history.

Glossary

Terminal
The window/app where you type commands.
Shell
The program inside the terminal that reads and runs your commands (bash, zsh, PowerShell, cmd).
Command Prompt (cmd)
Windows' older shell. Can't run PowerShell .ps1 scripts.
PowerShell
Windows' modern shell; its prompt starts with PS. Runs .ps1 scripts.
Directory
A folder. "Directory" is the technical term.
Current working directory
The folder your terminal is currently "standing in." Commands act here by default.
Path
An address for a file or folder. Relative = from where you are; absolute = the full address from the top.
Package / library
Reusable code someone else published that you install into your project.
pip
Python's package installer.
requirements.txt
A text list of the packages a project needs; install all with pip install -r.
Virtual environment (venv)
An isolated, per-project toolbox of Python packages.
PATH
The list of folders the shell searches to find programs you name.
Execution policy
A PowerShell security setting governing whether scripts may run.
Server
A program that waits for requests and responds — here, the app serving a web page.
localhost
"This computer." A server on localhost is reachable only from your own machine.
Port
A numbered channel on a machine (e.g. 5000) so multiple programs can share one address.
Digital signature
A tamper-proof stamp (the # SIG # block) proving a script is the genuine, unmodified file.
